# TwoPixel AGPL 合规清单（商业化场景）

本文件用于 TwoPixel 团队内部执行合规，不替代律师法律意见。

## 1. 结论先行

- 可以基于 AGPL-3.0 的 AstrBot 做修改、品牌替换、二次发布。
- 但合规义务通常不是“只公开改动片段”，而是向使用者提供该 AGPL 覆盖程序的对应源码（Corresponding Source）。
- 你们的商业化模块（注册登录、用户系统、计费、自研通信端、模型网关）可以不开源，前提是这些模块是独立系统，不是 AGPL 程序的衍生作品。

## 2. 需要开源 vs 可以闭源

### 2.1 需要开源的部分（AGPL 覆盖）

- twopixel-v2 中基于 AstrBot 的后端及其修改。
- 与该程序一起发布或部署所需的构建脚本、启动脚本、打包脚本。
- 你们对 AGPL 覆盖文件的改动（但不应只给 patch，建议提供完整可构建源码仓库）。

### 2.2 可以闭源的部分（可商业保留）

- 独立的账号系统/用户数据服务（独立仓库、独立部署、独立进程）。
- 独立的模型中转网关、计费系统、风控系统。
- 独立的私有通信服务（只通过网络 API 与 AGPL 程序交互）。
- 纯配置数据、密钥管理、运营后台（前提是未复制 AGPL 代码）。

## 3. 架构边界红线

- 不要把 AGPL 代码拷贝进闭源服务。
- 不要让闭源服务与 AGPL 代码形成同一程序的静态/动态链接关系。
- 使用“进程间网络调用”隔离：HTTP/gRPC/WebSocket API 交互优先。
- 如果闭源服务引用了 AGPL 代码文件或核心实现逻辑，可能触发衍生作品风险。

## 4. 对外提供服务时（SaaS）必须做的事

- 在 WebUI 或官网提供“源码获取入口”。
- 让使用该服务的用户可以拿到当前运行版本对应源码（建议提供 Git 仓库地址 + 对应 commit/tag）。
- 保留原许可证、版权声明、第三方 NOTICE。
- 明确你们的修改说明和版本标识（例如 `TwoPixel vX.Y.Z`）。

## 5. 发布与交付清单（每次上线前）

- 保留 AGPL License 文件与原作者版权信息。
- 在 README 增加：
  - 上游项目与许可证说明
  - TwoPixel 修改说明
  - 对应源码地址
- 在 UI 页脚或设置页增加“开源许可”入口。
- 版本发布时打 tag，并保留可复现构建步骤。
- Docker 发布时，同步公开该镜像对应源码版本和 Dockerfile。

## 6. 推荐仓库拆分

- `twopixel-oss`（开源）：AGPL 覆盖的 AstrBot 改造部分。
- `twopixel-private`（闭源）：账号、计费、模型网关、运营系统。
- 两者只通过 API 契约交互，不共享 AGPL 代码实现。

## 7. 当前项目落地建议

- 已做品牌替换文案可继续推进（TwoPixel CLI/UI）。
- 后续将“源码地址”放到：
  - WebUI 设置页
  - 登录页底部
  - README
- 增加一个公开文件：`THIRD_PARTY_NOTICES.md`，列出 AstrBot 与其他依赖许可证。

## 8. 团队执行规则（简版）

- 规则 A：凡是改 `twopixel-v2/astrbot/**` 的代码，默认进开源仓。
- 规则 B：凡是账号/计费/网关服务，禁止引用 `astrbot/**` 代码。
- 规则 C：上线必须有“源码入口 + 许可证信息 + 版本映射”。

## 9. 风险提示

- AGPL 合规是“法律 + 工程”双问题，最终以律师意见为准。
- 本清单用于工程落地与风险降低，不构成法律意见。
